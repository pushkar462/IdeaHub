03 · Decisions locked

Four architectural choices shape everything below.

Each was confirmed with the founder before any code moved. Every increment respects these — nothing here quietly re-litigates them.

Repository is truth
Loop stays a standalone Node + Express + Postgres app. Handbook references to bills / cases / partners tables are honoured as string refs on the post (linkedEntityId), never as CRM joins.
Groq stays
All AI features route through the existing Groq llama-3.3-70b client. The LOVABLE_API_KEY gateway swap is a future one-file change — the interface is ready in services/embedding/.
Engineering roles
The existing role enum (ADMIN, FOUNDER, FRONTEND, BACKEND, DEVOPS, AI_ML) is kept; routing is driven exclusively by the pre-existing SectionOwnership table.
Copy > routes
Handbook wording (“Post to Loop”, “Open / needs response”, LOOP-YYYY-NNNN numbering) is adopted verbatim on the surface. Internal URLs are unchanged — bookmarks survive.

04 · Phase 1 · the pilot

Eight increments that close the loop end to end.

These are the increments that take the app from “posts exist” to “the pilot definition of done in handbook C5 is met”. Every transition is DB-enforced, every resolution is captured, every question is findable, every hour of overdue SLA is visible.

I1
Shipped

State-machine trigger & post-number trigger

Handbook C2 · “every status transition is validated at the DB level”

A Postgres trigger enforces the Open → Discussing → Resolved state machine at the database, not just the service. A second trigger auto-generates LOOP-YYYY-NNNN numbers via a year-scoped sequence — concurrent inserts never collide. Illegal transitions raise LOOP_TRANSITION: exceptions that the error middleware surfaces as clean 400s.

Delivers
  • Postgres triggers generate_loop_post_number & loop_posts_transition_guard
  • Parked/Declined resolutions require a one-line reason at DB level
  • Friendly 400 mapping in Express error middleware
Key paths
migrations/20260708120000_state_machine_triggers · middleware/error.middleware.ts · controllers/post.controller.ts
I2
Shipped

Composer & Board copy pass, search-first UX

Handbook D1 & D3 · “before posting, search Loop”

The board defaults to Open / needs response. The composer opens with the handbook's own language, a three-pill type picker (Question / Problem / Idea), and a purple hint that names the canonical resolved post if one exists — catching duplicates at the moment of creation. Non-blocking soft-warn on unfamiliar bill / case / partner IDs.

Delivers
  • New needResponse query filter (OPEN + un-acknowledged)
  • Duplicate-check returns canonical excerpt for the top match
  • Shared lib/linkedEntity.ts for FE/BE format hints
Key paths
frontend/components/posts/CreatePostModal.tsx · pages/FeedPage.tsx · backend/services/intelligence/duplicate-check.service.ts
I3
Shipped

tsvector column + GIN index

Handbook C4 · “full-text search across title, body, and linked reference”

A generated searchVector tsvector column with weights A = title, B = linkedEntityId (simple config, no stemming, IDs stay literal), C = description. Feed search and duplicate-check both rank via plainto_tsquery + ts_rank through raw SQL, with an HNSW-style GIN index doing the heavy lifting.

Delivers
  • Generated tsvector stays synced automatically on every write
  • GIN index Post_searchVector_gin_idx for O(log n) recall
  • Prisma schema declares the column as Unsupported — no drift
Key paths
migrations/20260709120000_search_vector_gin · services/feed.service.ts · services/intelligence/duplicate-check.service.ts
I4
Shipped

Canonical answer block + GitHub build handoff

Handbook D2 & C6 · “the answer that stays, or the issue that ships”

When a Question is marked Answered, the resolver can inline a canonical answer that becomes a pinned comment hoisted into a green “✓ Canonical answer · marked by X” block above the thread. When a Problem or Idea is marked Fixed or Approved, an optional GitHub issue URL is captured and shown on the resolution card. Fixes a pre-existing bug where the status endpoint silently dropped resolution payload.

Delivers
  • New Post.buildIssueUrl column
  • Canonical answer landing as a comment with isCanonical=true — no field duplication
  • ResolveModal now conditionally renders per type × resolution
Key paths
migrations/20260709130000_post_build_issue_url · components/posts/ResolveModal.tsx · pages/PostDetailPage.tsx
I5
Shipped

SLA hardening + Loop health dashboard

Handbook B5 & C4 · “the black-hole rate is the health metric”

The hourly SLA cron now performs three guarded flips: OPEN 24–48h → AT_RISK silently, OPEN ≥ 48h → BREACHED with owner notification, DISCUSSING ≥ 7d without status change → BREACHED with notification. A new admin-only Loop health page surfaces the black-hole rate, per-section breakdown, time-to-acknowledge and time-to-resolve averages.

Delivers
  • Every cron flip guarded by updateMany — no double notifications
  • New GET /api/admin/loop-health aggregating counts + averages
  • Tailwind-only section-breakdown bar list (no chart lib pulled in)
Key paths
cron/sla.cron.ts · routes/admin.routes.ts · frontend/pages/AdminLoopHealthPage.tsx
I6
Shipped

Contributions view — factual layer

Handbook Section 10 · “recognition, not points”

Every profile now has a Contributions tab surfacing three panels — posts raised, posts helped resolve, questions answered — with an All time / Last 30d / Last 7d window toggle. Non-owners of another profile see only Recent posts; admin and founder see both. Sidebar renamed to My contributions.

Delivers
  • ?since=<ISO> filter on the existing endpoint, capped 100 items per list
  • Tabbed profile layout with per-tab window scoping
Key paths
backend/routes/user.routes.ts · frontend/pages/ProfilePage.tsx
I7
Shipped

Weekly digest endpoint (n8n)

Handbook Section 6 · “n8n owns the schedule, Loop owns the payload”

A new GET /api/digest/weekly returns the founder's Friday recap as JSON — new posts + top 3 by comments, resolved this week with resolver first name, awaiting Questions past 48h, stalled Ideas past 7d, and a single named recognition. Authenticated by an X-Digest-Token header. Empty token returns 503 by design so a misconfigured deploy can't leak.

Delivers
  • PII-light payload — first names only in named recognition
  • Every call logged on the backend; every rejection warns
  • README documents the payload shape and the n8n recipe
Key paths
backend/controllers/digest.controller.ts · routes/digest.routes.ts · config/env.config.ts

Delivers

05 · Phase 2 · the growth pass

Five landed, one deferred by design.

Phase 2 adds what only makes sense once the wider team is posting: voting to surface strong Ideas, campaigns to focus a month's attention, brand-manual typography, semantic-search infrastructure, and LLM assist for classification and KB promotion.

P1
Shipped

Voting on Ideas — loop_votes

Handbook C1 · “composite PK, one vote per user per post”

A new Vote table with composite primary key (postId, userId) enforces the “one vote per user” rule at the database level. The POST /api/posts/:id/vote endpoint toggles state and returns the new count. IdeaCard renders a proper ArrowBigUp button with a pressed state; the post store uses optimistic UI so votes feel instant even on slow networks.

Delivers
  • Composite-PK enforcement at DB level — race-safe
  • Feed & detail responses carry voteCount + hasVoted
  • Optimistic FE with revert-on-failure
Key paths
migrations/20260709160000_votes · services/feed.service.ts · frontend/components/ideas/IdeaCard.tsx

Shipped

pgvector semantic-search infrastructure

Handbook C4 aspirational · “retrieval-grounded, authorization-scoped”

The vector extension is enabled; posts carry a nullable 1536-dim embedding column indexed by HNSW with cosine ops. A pluggable EmbeddingProvider interface accepts a Noop (default, no external calls) or an OpenAI provider (text-embedding-3-small, env-gated). The Similar posts widget on post detail hides itself when the provider is disabled — ships silent until ops flips it.

Delivers

  • Zero-cost by default — noop provider skips all embed calls

P5
Shipped

Auto-tag composer hint + KB nominations

Handbook aspirational · “the model suggests; a human confirms”

On the composer, a debounced Groq classification returns a suggested type + section with a confidence and one-line reasoning — shown as a purple “Loop AI suggests” banner with Apply / Dismiss. Once the user picks anything manually, the classifier stops overriding. On the KB sweep page, admins can ask Groq to nominate resolved posts that look like they became rules; nominations render as a distinct card with a per-row Graduate button. The model never sets the flag itself.

Delivers
  • Strict enum validation collapses invalid model output to null
  • Same 15m / 20-req rate limit as other AI endpoints
  • Manual pick always wins — the AI is a hint, never an actor
Key paths
backend/services/intelligence/auto-tag.service.ts · services/intelligence/kb-nomination.service.ts · frontend/components/posts/CreatePostModal.tsx

06 · Verification

Six checks that prove every phase actually landed.

These run against the deployed app. The full 44-test walkthrough is in the session notes; the six below are the minimum “did this ship right” smoke test an engineer can hand to QA.

I1
New posts have numbers like LOOP-2026-0042. Illegal transitions fail at DB level:
UPDATE "Post" SET status = 'RESOLVED' WHERE id = <open post id>;
-- expects: LOOP_TRANSITION: Resolution is required…
I3
Search plan uses the GIN index:
EXPLAIN SELECT id FROM "Post"
 WHERE "searchVector" @@ plainto_tsquery('english', 'bill');
-- expects: Bitmap Index Scan on Post_searchVector_gin_idx
I7
Digest endpoint returns JSON for the correct token, 401 for anything else:
curl -H "X-Digest-Token: $DIGEST_TOKEN" \
     https://<backend>/api/digest/weekly
P1
Vote uniqueness enforced at DB level:
SELECT COUNT(*) FROM "Vote" GROUP BY "postId", "userId"
 HAVING COUNT(*) > 1;
-- expects: zero rows
P4
Extension and column both live on the DB:
SELECT extname FROM pg_extension WHERE extname = 'vector';
SELECT indexname FROM pg_indexes
 WHERE indexname = 'Post_embedding_hnsw_idx';
P5
In the composer, paste a long title. Within about a second, the purple Loop AI suggests banner surfaces a type and section with a reasoning line, and applying it flips both fields. Manual picks always beat the classifier.
What breaks the pilot. Only three states are genuinely broken: a post creates without a number (I1 migration didn't run), the composer never suggests a type (Groq key rejected or rate-limited), or the SLA cron never advances state (the Render service was never restarted after a schedule change). Everything else fails gracefully — empty widgets, silent 401s, or noop AI — without user-visible errors.

07 · Deploy sequence

Three steps, every time.

  1. Push to main. Vercel picks up the frontend automatically; Render picks up the backend automatically.
  2. Run migrations against Supabase from your laptop. The Shared Pooler URL on port 5432 works with prisma migrate deploy; the transaction pooler on 6543 does not.
  3. Set environment variables in Render. The digest endpoint needs DIGEST_TOKEN before it will respond to n8n; embeddings need EMBED_PROVIDER=openai + OPENAI_API_KEY if and when you turn them on.
